# 智能体问题排查与修复说明

本文档记录 2026-04-04 在“校园交换机智能体节能控制台”中排查到的智能体相关问题、根因和对应修复方式，便于后续演示、答辩和运维交接。

## 1. DeepSeek Key 已接入，但智能体仍然报 401

### 现象

- 前端已经填写 `deepseek-reasoner`
- 智能体巡检直接失败
- 后端日志出现 `401 Incorrect API key provided`

### 根因

- 只配置了模型名和 API Key，没有把兼容服务地址切到 DeepSeek
- 请求仍然打到了 OpenAI 默认地址

### 修复方式

- 增加兼容服务地址识别逻辑
- 当模型名以 `deepseek` 开头，且用户未手填地址时，自动推断为 `https://api.deepseek.com`
- 前端和后端都统一按兼容模式保存和读取模型接入配置

### 结果

- DeepSeek 配置不再因为漏填 Base URL 而直接 401

## 2. DeepSeek 返回 `does not support this tool_choice`

### 现象

- 自治巡检失败
- 作业错误信息为 `400 deepseek-reasoner does not support this tool_choice`

### 根因

- 旧逻辑仍有一部分请求路径会带上 `tool_choice`
- DeepSeek 的兼容接口不接受这类调用方式

### 修复方式

- 将 DeepSeek 兼容模式统一切到 `chat.completions`
- 避开 `Responses API + tool_choice` 这条不兼容路径
- 同时保留 OpenAI 官方模型的 `Responses API` 路径

### 结果

- 修复后，新产生的作业中不再出现新的 `tool_choice` 报错

## 3. DeepSeek 多轮工具调用过程中上下文续传不完整

### 现象

- 智能体能够调用部分工具
- 但多轮推理过程中容易中断或行为异常

### 根因

- 在 DeepSeek thinking mode 下，多轮工具调用需要把上一轮 assistant message 中的 `reasoning_content` 一并回传
- 旧实现只回传了 `content` 和 `tool_calls`

### 修复方式

- 新增 assistant continuation message 组装逻辑
- 在多轮 `chat.completions` 调用中保留 `reasoning_content`

### 结果

- DeepSeek 多轮工具调用链路更稳定

## 4. 规划智能体会不断调用工具，但始终不提交最终计划

### 现象

- 作业里可以看到 `get_console_snapshot`、`list_interfaces`、`estimate_port_action`
- 但最后报 `智能体未能生成有效计划，请稍后重试`

### 根因

- 模型完成了信息采集，但没有在限定轮次内主动提交最终计划
- 达到最大工具轮次后，原逻辑直接报错

### 修复方式

- 增加规划阶段 `finalize` 收束逻辑
- 当模型未直接提交终稿时，转入“JSON 终稿整理”
- 若 JSON 终稿仍失败，则根据已完成的动作估算结果生成一份保守计划

### 结果

- 规划阶段不再因为“没主动交终稿”而直接失败
- 最新作业已经验证可正常进入 `finalize`

## 5. 风险评审智能体也依赖函数提交，稳定性不足

### 现象

- 规划成功后，评审阶段仍可能因为结构化输出不稳定而失败

### 根因

- 评审阶段其实不需要外部工具
- 旧实现仍依赖函数调用形式返回结论，兼容性不如直接 JSON 输出

### 修复方式

- 将 DeepSeek 兼容模式下的评审阶段改为 JSON 输出
- 增加本地保守评审兜底
- 即使模型评审输出不稳定，也能基于风险评分生成保守门控结论

### 结果

- 评审阶段可以稳定完成
- 最新作业已验证能输出“需谨慎 / 人工审批”结论

## 6. 多个后端实例同时运行，导致同一数据库被重复写入

### 现象

- 数据库里同一时间出现互相矛盾的作业结果
- 有的作业走新逻辑，有的作业仍报旧错误

### 根因

- 机器上同时存在多个后端进程
- 它们共享同一套 SQLite 数据库
- 其中有旧实例仍在按旧逻辑执行自治巡检

### 修复方式

- 清理重复后端进程，只保留一个最新实例
- 为自治巡检增加数据库租约锁，避免多实例重复跑同一轮巡检

### 结果

- 现在只保留一个后端实例写数据库
- 新作业不再出现“新旧逻辑混跑”的情况

## 7. 后端异常退出后，数据库里残留“运行中作业”

### 现象

- 页面会一直显示某个智能体作业还在运行
- 但对应后端实例其实早已停止

### 根因

- 旧实例退出时没有完成作业收尾
- SQLite 中仍保留 `status = running`

### 修复方式

- 服务重启时自动扫描 `running` 作业
- 将陈旧运行中作业统一标记为失败并写入终止说明
- 同时手动清理了一次历史残留作业

### 结果

- 重启后不会再长期保留假运行状态

## 8. 本次最终落地的关键代码位置

- DeepSeek 兼容与规划/评审收束逻辑：
  - `server/src/services/openai-agent.ts`
- 自治巡检单实例锁：
  - `server/src/services/agent-autonomy.ts`
  - `server/src/services/interface-db.ts`
- 运行中作业自恢复：
  - `server/src/services/agent-runtime.ts`

## 9. 当前验证结果

### 已验证通过

- 最新后端实例只剩一个
- 新的 DeepSeek 作业不再出现 `tool_choice` 报错
- 规划阶段已成功进入 `finalize`
- 评审阶段已成功输出结构化结论
- 最新两轮自治作业都能正常收束为 `skipped`，不再以异常失败结束

### 当前表现说明

- 最新作业之所以显示 `skipped`，不是智能体崩了
- 而是规划或评审后判断“本轮没有符合自动执行条件的动作”，因此系统主动不执行

## 10. 后续建议

- 如果后面要继续提升 DeepSeek 质量，可以再补两项：
  - 记录规划阶段最终 assistant 原文，便于进一步分析模型行为
  - 将当前保守计划兜底结果单独标记为 `fallback`，方便在前端区分“模型直出”和“模型收束后保守生成”
